# 📌 Casos Especiales y Errores Comunes con `Number`

## 📖 Definición

Los números en JavaScript están basados en el estándar **IEEE 754 (doble precisión, 64 bits, coma flotante)**.  
Esto significa que, aunque son muy versátiles, presentan **limitaciones y comportamientos inesperados** cuando se trabajan con decimales, grandes cantidades o conversiones.  
Estos casos especiales son **importantes de conocer** para evitar errores en cálculos financieros, validaciones y lógica de programas.

---

## 📌 Sintaxis general

No existe una sintaxis específica para estos errores:  
Aparecen cuando se realizan operaciones numéricas.

Ejemplo:

```js
console.log(0.1 + 0.2); // 0.30000000000000004
console.log(Number.isNaN(NaN)); // true
console.log(9999999999999999); // 10000000000000000
```

---

## ⚙️ Características principales

1. **Precisión limitada**: solo se pueden representar exactamente enteros hasta `2^53 - 1`.
    
2. **Errores con decimales**: operaciones con coma flotante pueden dar resultados aproximados.
    
3. **NaN especial**: "Not-a-Number" representa un error numérico, pero tiene comportamientos peculiares.
    
4. **Infinity**: operaciones que superan el límite superior retornan `Infinity` o `-Infinity`.
    
5. **Conversión automática**: algunos métodos convierten valores inesperadamente (ej. `parseInt("abc")`).
    

---

## 🔥 Casos especiales

### 1. Errores de precisión decimal

```js
console.log(0.1 + 0.2); // 0.30000000000000004
console.log(0.3 === 0.1 + 0.2); // false
```

📍 Esto sucede porque los decimales **no se representan exactamente** en binario.

✅ **Solución**

```js
function sonIguales(a, b) {
  return Math.abs(a - b) < Number.EPSILON;
}

console.log(sonIguales(0.1 + 0.2, 0.3)); // true
```

---

### 2. Límite de enteros seguros

```js
console.log(Number.MAX_SAFE_INTEGER); // 9007199254740991
console.log(9007199254740991 + 1); // 9007199254740992
console.log(9007199254740991 + 2); // 9007199254740992 (¡Error!)
```

✅ **Solución**

- Usar `Number.isSafeInteger()` para verificar.
    
- Usar `BigInt` para cálculos grandes:
    

```js
let big = 9007199254740991n + 2n;
console.log(big); // 9007199254740993n
```

---

### 3. `NaN` (Not-a-Number)

```js
console.log(NaN === NaN); // false
console.log(Number.isNaN(NaN)); // true
```

❌ **Error común**

```js
isNaN("hola"); // true (convierte "hola" → NaN)
```

✅ **Uso correcto**

```js
Number.isNaN("hola"); // false
Number.isNaN(NaN);    // true
```

---

### 4. Infinity y -Infinity

```js
console.log(1 / 0);  // Infinity
console.log(-1 / 0); // -Infinity
console.log(Infinity > 1000000000); // true
```

📍 Aunque existen, no son valores prácticos para cálculos, salvo en comparaciones o validaciones.

---

### 5. Conversiones inesperadas

```js
console.log(Number.parseInt("123abc")); // 123
console.log(Number.parseInt("abc123")); // NaN
console.log(Number.parseFloat("3.14m")); // 3.14
```

📍 `parseInt` y `parseFloat` leen hasta donde detectan un número válido.

---

## ✅ Buenas prácticas

1. Para cálculos financieros, evitar `Number` → usar librerías de precisión decimal o `BigInt`.
    
2. Para comparar decimales, usar `Number.EPSILON`.
    
3. Validar siempre con `Number.isNaN`, no con `isNaN` global.
    
4. Antes de operar con enteros grandes, usar `Number.isSafeInteger`.
    
5. Manejar explícitamente los casos de `Infinity` y `NaN` en validaciones.
    

---

## 📝 Resumen

- Los números en JS tienen **precisión limitada** por IEEE 754.
    
- **Errores comunes**: decimales inexactos, pérdida de precisión con enteros grandes, `NaN` especial, `Infinity`.
    
- **Herramientas clave**: `Number.EPSILON`, `Number.isNaN`, `Number.isSafeInteger`, `BigInt`.
    
- **Buenas prácticas**: no confiar en la igualdad directa de decimales, validar siempre antes de operar y evitar cálculos críticos con `Number` puro.
