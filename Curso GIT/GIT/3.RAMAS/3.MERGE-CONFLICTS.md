
## GIT – MERGE CONFLICTS (con VS Code y log visual)

### ¿Qué es un conflicto de merge?

Un **conflicto de merge** ocurre cuando Git no puede decidir automáticamente qué cambios conservar al unir dos ramas, porque **ambas ramas editaron la misma parte del mismo archivo** de forma diferente.

Git detiene el proceso de fusión para que una persona lo resuelva manualmente.

---

## ¿Cuándo aparece un conflicto?

Un conflicto ocurre al usar `git merge nombre-rama` si:

- Ambas ramas **modificaron la misma línea de código**.
    
- Una rama **eliminó una parte** que la otra **modificó**.
    
- Se hicieron **cambios incompatibles** en el mismo archivo.
    

---

## Ver el historial y los commits implicados

Antes o después de un merge, es útil visualizar el historial de ramas con:

```bash
git log --oneline --all --graph
```

Este comando:

- Muestra los commits de **todas las ramas**.
    
- Usa una vista en forma de **árbol (graph)**.
    
- Ayuda a ver **dónde se separaron y hacia dónde se están fusionando** las ramas.
    

**Ejemplo de salida:**

```text
* 5d9f1b9 (HEAD -> main) último commit en main
| * 3c4e211 (login) cambios en login
|/
* b1a2d3c commit común entre ambas
```

Esto ayuda a entender **qué se va a fusionar y desde dónde**.

---

## ¿Cómo se ve un conflicto en el archivo?

Git edita el archivo afectado con marcas visuales como estas:

```html
<<<<<<< HEAD
<p>Hola desde la rama actual</p>
=======
<p>Hola desde la rama que quiero fusionar</p>
>>>>>>> login
```

Esto marca:

- `<<<<<<< HEAD`: lo que existe en la rama actual.
    
- `=======`: separación entre versiones.
    
- `>>>>>>> login`: lo que viene de la rama que estás fusionando (`login`).
    

---

## Resolver conflictos con VS Code

Cuando se abre VS Code con un archivo en conflicto, se presentan **botones de acción automática** encima del archivo.

### Opciones visuales que muestra VS Code:

- **Accept Current Change**: conservar lo que está en la rama actual (`HEAD`).
    
- **Accept Incoming Change**: conservar lo que viene de la rama que estás fusionando.
    
- **Accept Both Changes**: unir ambas versiones en el archivo.
    
- **Compare Changes**: ver en detalle las diferencias antes de decidir.
    
 Consejo: “Both Changes” no siempre es la mejor opción si ambas versiones hacen lo mismo de diferente forma. Revisá qué conviene conservar.

---

## Resolución paso a paso

### 1. Ejecutar el merge

```bash
git switch main
git merge login
```

Si hay conflicto, Git lo avisa en consola:

```text
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.
```

---

### 2. Ver los archivos en conflicto

```bash
git status
```

Git listará los archivos así:

```text
Unmerged paths:
  both modified:   index.html
```

---

### 3. Resolver el archivo

Abrir el archivo en VS Code. Aparecen las **opciones visuales** arriba de cada bloque en conflicto.

- Elegir una opción (por ejemplo, "Accept Incoming Change").
    
- Editar manualmente si es necesario.
    
- Guardar el archivo.
    

---

### 4. Marcar como resuelto

```bash
git add index.html
```

Esto le dice a Git que ya se resolvió ese conflicto.

---

### 5. Finalizar el merge

```bash
git commit
```

Se crea un commit de tipo `Merge: abc123 def456`. Podés dejar el mensaje por defecto o personalizarlo.

---

## Abortando un merge (si no querés continuar)

Si por algún motivo se desea **cancelar el merge y volver al estado anterior**:

```bash
git merge --abort
```

Este comando **deshace todo el proceso de fusión** si aún no se resolvieron los conflictos.

---

## Resumen visual

```text
Rama actual:         HEAD (main)
Rama a fusionar:     login

Resultado:
- Se mezclan los commits si no hay conflictos.
- Si hay conflictos → Git los marca.
- Se editan → se agregan (`git add`) → se confirma (`git commit`).
```

## Comandos clave

|Acción|Comando|
|---|---|
|Ver ramas y estructura visual|`git log --oneline --all --graph`|
|Iniciar merge|`git merge nombre-de-la-rama`|
|Ver archivos en conflicto|`git status`|
|Cancelar el merge|`git merge --abort`|
|Marcar archivo como resuelto|`git add archivo`|
|Finalizar el merge|`git commit`|
