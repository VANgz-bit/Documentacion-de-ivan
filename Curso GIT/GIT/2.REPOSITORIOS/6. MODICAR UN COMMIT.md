
## ¿Qué es `git commit --amend`?

Es una opción del comando `git commit` que **modifica el último commit realizado**. Sirve para:

- Cambiar el **mensaje** del último commit.
    
- Agregar **nuevos archivos** o **modificaciones** al último commit.
    
- Corregir errores sin crear un nuevo commit.
    

El commit anterior **es reemplazado por uno nuevo**, así que **técnicamente se elimina el commit anterior y se crea otro nuevo con un nuevo hash**.

---

## ¿Qué NO hace?

- No se puede usar para modificar **commits anteriores al último** (para eso se usan otras herramientas como `rebase`).
    
- No se debe usar si **ya se empujó el commit a un repositorio remoto compartido**, salvo que sepas lo que haces (porque se reescribe el historial).
    

---

## Situaciones típicas donde se usa `--amend`

1. **Te olvidaste de agregar un archivo antes de hacer commit.**
    
2. **Querés corregir el mensaje del commit.**
    
3. **Cometiste un error menor justo después del último commit.**
    

---

## PASO A PASO CON CADA CASO

---

### 🟩 1. Cambiar el mensaje del último commit (sin tocar los archivos)

```bash
git commit --amend
```

#### ¿Qué pasa?

- Abre tu editor de texto por defecto (por ejemplo, Vim o VS Code).
    
- Te muestra el mensaje actual del commit.
    
- Podés editarlo, guardarlo y cerrarlo.
    

#### Resultado:

- Git crea **un nuevo commit con el mismo contenido**, pero **con el nuevo mensaje**.
    
- El anterior se elimina (internamente).
    

#### ⚠️ Cuidado:

Esto cambia el **hash del commit**. No se debe hacer si ese commit ya fue enviado a un remoto compartido (ej: GitHub).

---

### 🟨 2. Te olvidaste de agregar un archivo antes de hacer commit

#### Supongamos que te olvidaste de agregar `script.js`:

```bash
git add script.js
git commit --amend
```

#### ¿Qué pasa?

- Git **agrega el nuevo archivo al área de staging**, y con `--amend`, lo **incluye en el último commit como si siempre hubiese estado allí**.
    
- Se abre el editor para que puedas cambiar (o no) el mensaje del commit.
    
- Si no cambiás el mensaje y solo guardás/cerrás, **el mensaje quedará igual** pero el contenido del commit se actualiza.
    

---

### 🟦 3. Modificar archivos existentes después de un commit

#### Supongamos que ya hiciste un commit, pero querés corregir algo en `index.html`:

```bash
nano index.html  # o el editor que uses
# hacés la corrección

git add index.html
git commit --amend
```

- Git incluye el nuevo cambio dentro del commit anterior.
    
- Se abre el editor para el mensaje.
    

---

## ¿Y si solo quiero cambiar el mensaje, sin abrir el editor?

Podés hacerlo directamente con la opción `-m`:

```bash
git commit --amend -m "Nuevo mensaje corregido del commit"
```

---

## ¿Qué pasa con el historial?

- El commit anterior desaparece del historial.
    
- Se crea uno nuevo con nuevo hash.
    
- Si alguien ya tenía el commit anterior (por ejemplo, en GitHub), hay conflictos al hacer `push`.
    

---

## ¿Y si ya hice `push` al remoto y hago un `--amend`?

### 1. Git te avisa que no podés hacer `push` directo:

```bash
git push
# error: non-fast-forward
```

### 2. Solución (con precaución): usar `--force` o `--force-with-lease`:

```bash
git push --force
# o mejor:
git push --force-with-lease
```

#### ⚠️ Advertencia:

Esto sobrescribe el historial remoto. No lo uses en ramas compartidas sin hablar con los demás colaboradores. Puede causar pérdida de trabajo si alguien está trabajando en la versión anterior del commit.

---

## Casos de uso típicos + comandos

|Situación|Comando|
|---|---|
|Cambiar mensaje del commit|`git commit --amend`|
|Agregar archivos olvidados|`git add archivo && git commit --amend`|
|Corregir cambios después del commit|`git add archivo && git commit --amend`|
|Cambiar mensaje sin editor|`git commit --amend -m "Nuevo mensaje"`|
|Subir commit corregido (peligroso)|`git push --force-with-lease`|

---

## ¿Cómo saber si puedo usar `--amend` sin problemas?

- Si **NO hiciste `push` aún**: podés usar `--amend` libremente.
    
- Si **YA hiciste `push`**:
    
    - **Si trabajás solo en esa rama**, podés usar `--force-with-lease`.
        
    - **Si es una rama compartida**, **consultá antes** o evitá modificar el historial.
        

---

## BONUS: ¿Qué editor se abre con `--amend`?

Depende de tu configuración. Git usa por defecto `Vim`, pero podés cambiarlo:

```bash
git config --global core.editor "code --wait"     # Para Visual Studio Code
git config --global core.editor "nano"            # Para Nano
```
