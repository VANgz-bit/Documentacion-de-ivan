--
Git permite un control total sobre los archivos versionados. A través de comandos como `checkout`, `restore`, `reset`, `mv` (para renombrar) y `status`, es posible **ver el estado del proyecto, deshacer cambios, cambiar ramas, y modificar el nombre de archivos**, todo de manera precisa y segura.

---

##  `git status`: Ver el estado de los archivos

### **¿Qué hace?**

Muestra en qué estado están los archivos dentro del repositorio:

- Cuáles están modificados.
    
- Cuáles están preparados para el próximo commit.
    
- Cuáles no están siendo rastreados por Git.
    
- En qué rama se está trabajando.
    

### **Sintaxis:**

```bash
git status
```

### **Ejemplo de salida:**

```bash
On branch main
Changes not staged for commit:
  modified:   index.html

Untracked files:
  nuevo-archivo.txt
```

### **Estados posibles:**

|Estado|Descripción|
|---|---|
|**Untracked**|El archivo existe pero Git no lo está siguiendo (aún no se hizo `git add`).|
|**Modified**|El archivo fue modificado, pero no está en el área de preparación.|
|**Staged (en verde)**|El archivo está en la zona de preparación, listo para el próximo commit.|
|**Committed**|El archivo ya fue registrado en el historial (último commit).|

---

##  `git checkout`: Cambiar de ramas o recuperar archivos

### **¿Qué hace?**

Tiene **dos funciones principales**:

1. Cambiar de rama.
    
2. Recuperar la versión anterior de un archivo desde el último commit.
    

> ⚠️ _Desde Git 2.23, se recomienda usar `git switch` y `git restore` en lugar de `checkout` para tareas específicas. `checkout` sigue funcionando pero es más general y menos intuitivo._

### **Cambiar de rama:**

```bash
git checkout nombre-de-la-rama
```

Esto cambia el HEAD del repositorio a esa rama y actualiza los archivos de trabajo con su contenido.

### **Revertir un archivo a su último commit:**

```bash
git checkout -- archivo.txt
```

Esto descarta todos los cambios locales **no preparados** en ese archivo y lo deja como estaba en el último commit.

---

##  `git restore`: Deshacer cambios en archivos (más moderno que checkout)

### **¿Qué hace?**

Es el comando moderno (desde Git 2.23) para **deshacer cambios locales** en el área de trabajo o en la zona de preparación. Está enfocado en archivos, no en ramas.

### **Restaurar archivo desde el último commit (descartar cambios locales):**

```bash
git restore archivo.txt
```

Esto reemplaza el archivo actual con la versión del último commit.

### **Quitar archivo del staging (sin deshacer el cambio):**

```bash
git restore --staged archivo.txt
```

Esto **saca el archivo del área de preparación**, pero **mantiene los cambios en el área de trabajo**.

---

##  `git reset`: Manipular el historial y deshacer preparaciones

### **¿Qué hace?**

`git reset` es un comando potente que puede:

- Quitar archivos del staging.
    
- Revertir commits recientes.
    
- Mover el HEAD del repositorio.
    

Existen **3 modos principales**, dependiendo del nivel de agresividad:

---

### **1. `git reset --soft`**

- Mueve el HEAD a un commit anterior.
    
- **Los cambios siguen en staging** listos para hacer un nuevo commit.
    

```bash
git reset --soft HEAD~1
```

Esto deshace el último commit pero deja los archivos listos para hacer uno nuevo.

---

### **2. `git reset --mixed`** (modo por defecto)

- Mueve el HEAD.
    
- **Quita archivos del staging**, pero deja los cambios en el área de trabajo.
    

```bash
git reset HEAD archivo.txt
# o
git reset --mixed HEAD~1
```

---

### **3. `git reset --hard`** ⚠️ CUIDADO

- Mueve el HEAD.
    
- **Quita los archivos del staging y descarta los cambios del área de trabajo.**
    
- Deja el proyecto tal como estaba en el commit indicado.
    

```bash
git reset --hard HEAD~1
```

⚠️ Este comando **borra definitivamente los cambios**, no se pueden recuperar fácilmente.

---

##  Cambiar el nombre de un archivo: `git mv`

### **¿Qué hace?**

Renombra un archivo y notifica a Git del cambio, evitando que lo interprete como eliminación + creación.

### **Sintaxis:**

```bash
git mv viejo-nombre.txt nuevo-nombre.txt
```

Esto mueve el archivo y lo deja automáticamente en el área de preparación.

### **Equivalente manual:**

```bash
mv viejo.txt nuevo.txt
git add nuevo.txt
git rm viejo.txt
```

Pero `git mv` es más limpio y automático.

---

##  Comparación rápida: ¿Qué hace cada uno?

|Comando|Propósito principal|Afecta área de trabajo|Afecta staging|Afecta historial|
|---|---|---|---|---|
|`git status`|Ver en qué estado están los archivos|No|No|No|
|`git checkout -- archivo`|Recuperar archivo desde último commit (descarta cambios locales)|Sí|No|No|
|`git restore archivo`|Igual que arriba, pero moderno|Sí|No|No|
|`git restore --staged`|Quitar archivos del staging sin perder los cambios|No|Sí|No|
|`git reset`|Mover HEAD o quitar archivos del staging|Depende del modo|Sí|Sí (si hay commits)|
|`git mv`|Renombrar archivo y registrarlo|Sí|Sí|No|

---

##  Conclusión general

Git ofrece un **conjunto poderoso de herramientas** para manejar el ciclo completo de cambios en un proyecto:

- `git status` permite entender claramente el estado actual de cada archivo.
    
- `git checkout` y `git restore` ayudan a **deshacer cambios sin comprometer el historial**.
    
- `git reset` permite modificar el historial o deshacer etapas del flujo (preparación o commits).
    
- `git mv` permite renombrar archivos correctamente sin perder el seguimiento de Git.
    

Dominar estos comandos es esencial para tener control total sobre el proyecto, evitar errores y trabajar de forma segura tanto en solitario como en equipos.
