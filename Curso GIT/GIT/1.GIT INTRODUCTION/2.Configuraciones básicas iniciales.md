
## ** Configuración inicial de Git**

Cuando instalamos Git por primera vez, hay que configurarlo para que sepa **quién sos** (nombre y correo), con qué editor trabajar, qué comportamiento usar por defecto, entre otros detalles.

Git organiza sus configuraciones en tres **niveles** o **ámbitos**:

---

### **1. Configuración Global (`--global`)**

**¿Qué es?**  
Es una configuración que **aplica para todos los proyectos Git** que tengas en tu computadora. Se guarda en un archivo que Git crea en tu carpeta personal (`~/.gitconfig` en Linux/Mac o `C:\Users\TuNombre\.gitconfig` en Windows).

**¿Para qué sirve?**  
Para definir tus datos de identidad, editor predeterminado, colores, alias y más. Es lo primero que deberías configurar al instalar Git.

**Ejemplos importantes:**

```bash
git config --global user.name "Iván Emmanuel Gómez"
git config --global user.email "ivan@example.com"
git config --global core.editor "code --wait"        # Usa VS Code como editor
git config --global color.ui auto                     # Activa colores en consola
git config --global init.defaultBranch main           # Usa "main" como rama por defecto
```

**Resultado:**  
Git usará estos datos por defecto **en todos los proyectos**, a menos que algún proyecto tenga su propia configuración local.

---

### **2. Configuración Local (por proyecto)**

**¿Qué es?**  
Es la configuración **específica de un repositorio Git en particular**. Se guarda dentro de la carpeta `.git/config` del proyecto.

**¿Para qué sirve?**  
Si necesitás cambiar alguna configuración para un proyecto en especial, lo hacés a nivel local y **no afecta a los demás proyectos**.

**Ejemplos típicos:**

```bash
git config user.name "Nombre Distinto"
git config user.email "otro@email.com"
```

Esto **anula la configuración global**, pero **solo dentro de ese repositorio**.

También podemos cambiar cosas como el editor, merge tool, push predeterminado, etc., solo para ese proyecto.

---

### **3. Configuración del servidor remoto (GitHub, GitLab, etc.)**

**¿Qué es?**  
Es la configuración que define **con qué servidor remoto** (como GitHub) va a trabajar tu repositorio.

**¿Dónde se ve esto?**  
Está dentro de `.git/config` o se agrega con el comando `git remote`.

**Ejemplos comunes:**

```bash
git remote add origin https://github.com/tuusuario/repositorio.git
git remote -v   # Muestra las URLs remotas configuradas
```

También podés configurar autenticación, tokens, claves SSH y más para trabajar con servidores remotos de forma segura.

---

## **¿Cómo ver la configuración actual?**

Podés consultar la configuración de Git en diferentes niveles con:

```bash
git config --list                # Muestra la configuración combinada (global + local)
git config --global --list       # Solo la global
git config --local --list        # Solo la local (si estás en un repo)
```

---

## **Resumen de niveles**

|Nivel|Afecta a...|Archivo de configuración|Comando|
|---|---|---|---|
|Global|Todos tus proyectos|`~/.gitconfig`|`git config --global`|
|Local|Solo un proyecto|`./.git/config`|`git config` (sin --global)|
|Server|Servidor remoto (GitHub, GitLab, etc.)|definido en `.git/config` vía `git remote`|`git remote add` / `set-url`|

---

## ** Para practicar**

Después de configurar Git con tu nombre y correo, probá esto:

1. Creá una carpeta nueva
    
2. Iniciá un repo con `git init`
    
3. Hacé un `git config --local --list`
    
4. Observá cómo se combinan las configuraciones globales con las locales
    
5. Luego agregá un remoto con `git remote add` (aunque sea ficticio)

## **🧱 `core.autocrlf`: Control de saltos de línea en Git**

### **¿Por qué importa esto?**

Diferentes sistemas operativos **usan distintos caracteres para los saltos de línea**:

- **Linux / macOS**: usan `LF` (Line Feed)
    
- **Windows**: usa `CRLF` (Carriage Return + Line Feed)
    

Este detalle puede parecer menor, pero si trabajás en equipo (uno con Windows y otro con Linux o macOS), **Git puede detectar cambios falsos** en archivos solo por diferencias en los saltos de línea.

---

## **¿Qué es `core.autocrlf`?**

Es una configuración que le indica a Git **cómo debe convertir los saltos de línea al guardar o leer archivos**, dependiendo del sistema operativo.

---

### **Modos posibles de `core.autocrlf`**

|Valor|¿Qué hace?|¿Cuándo usarlo?|
|---|---|---|
|`true`|**Convierte CRLF a LF al guardar en el repositorio** y **LF a CRLF al extraer en tu máquina**|Si usás **Windows**|
|`input`|**Convierte CRLF a LF al guardar**, pero **no cambia nada al extraer**|Si usás **Linux o macOS**|
|`false` (por defecto)|No hace ninguna conversión automática|Solo si **vos controlás manualmente** los saltos de línea|

---

### **Ejemplos según tu sistema:**

- **Si estás en Windows** (recomendado):
    
    ```bash
    git config --global core.autocrlf true
    ```
    
- **Si estás en Linux o macOS** (recomendado):
    
    ```bash
    git config --global core.autocrlf input
    ```
    

---

### **¿Dónde se guarda esta configuración?**

Como todo lo global, se guarda en tu archivo:

- `~/.gitconfig` (Linux/macOS)
    
- `C:\Users\TuUsuario\.gitconfig` (Windows)
    

---

### **¿Cómo saber si lo tenés activo?**

```bash
git config --global core.autocrlf
```

Si no devuelve nada, entonces **no está configurado aún**.

---

### **Consejo extra: controlar diferencias con `.gitattributes`**

Cuando trabajás en equipo, es buena práctica agregar un archivo llamado `.gitattributes` para indicar explícitamente cómo deben tratarse los archivos:

```txt
* text=auto
```

Eso le dice a Git que trate todos los archivos de texto con detección automática y evite conflictos innecesarios de saltos de línea.

---

##  Resumen 

Agregando esto a lo que ya vimos:

- **`core.autocrlf`**  ayuda a evitar problemas de compatibilidad de saltos de línea entre sistemas.
    
- Es una configuración fundamental si trabajamos en Windows, o en equipos con sistemas distintos.
    
- Se debe definir a nivel global según tu sistema operativo.
    
- También podés usar `.gitattributes` para asegurarte de que todos en el equipo usen el mismo formato.
