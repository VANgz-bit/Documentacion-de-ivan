
## ¿Qué es `git pull`?

`git pull` es el comando que se utiliza para **descargar los cambios desde un repositorio remoto** (como GitHub) y **actualizar automáticamente tu rama local** con esos cambios.

> Es como una combinación de dos comandos:  
> `git fetch` (descargar) + `git merge` (fusionar).

---

## ¿Para qué se usa?

Se usa para mantener tu repositorio local **actualizado** con los cambios que se hayan subido al repositorio remoto por vos u otros colaboradores.

---

## Sintaxis básica

```bash
git pull <remoto> <rama>
```

### Ejemplo común:

```bash
git pull origin main
```

- `origin`: es el nombre del repositorio remoto (por defecto).
    
- `main`: es la rama que estás actualizando.
    

Si ya configuraste un “upstream” con `git push -u`, podés hacer simplemente:

```bash
git pull
```

---

## ¿Qué hace internamente?

1. Conecta con el repositorio remoto (`origin`).
    
2. Descarga todos los commits que no tenés.
    
3. Intenta **fusionar automáticamente** los cambios en tu rama local.
    

---

## Flujo típico de uso

1. Estás trabajando localmente.
    
2. Antes de empezar un cambio importante o hacer `git push`, ejecutás:
    
    ```bash
    git pull origin main
    ```
    
3. Así evitás conflictos con otros cambios que ya estén en GitHub.
    

---

## Variantes útiles

### `git pull --rebase`

```bash
git pull --rebase origin main
```

En vez de hacer un merge, **reordena tus commits locales encima de los nuevos commits del servidor**. Esto:

- Mantiene un historial más limpio y lineal.
    
- Evita los commits de merge innecesarios.
    

---

## ¿Qué pasa si hay conflictos?

Si los cambios del remoto y tus cambios locales **editan las mismas líneas** de código, Git no puede decidir cuál usar. Entonces:

1. Git te avisa que hay un conflicto.
    
2. Tenés que abrir los archivos afectados y resolverlos manualmente.
    
3. Una vez resuelto, hacés:
    
    ```bash
    git add archivo-afectado
    git commit    # si fue un merge
    # o
    git rebase --continue   # si fue con rebase
    ```
    

---

## Diferencia entre `pull`, `fetch` y `merge`

|Comando|Qué hace|
|---|---|
|`git fetch`|Solo descarga los cambios remotos, **no los aplica**.|
|`git merge`|Fusiona ramas (usado luego de fetch).|
|`git pull`|Hace ambos pasos juntos: fetch + merge.|

---

## Posibles errores comunes

### “Your branch is behind”

- Significa que tu rama local está desactualizada.
    
- Solución: hacé un `git pull`.
    

### “Merge conflict”

- Git no puede fusionar automáticamente.
    
- Solución: resolver el conflicto manualmente y luego continuar.
    

### “You have unstaged changes”

- Tenés cambios sin confirmar y no podés hacer pull.
    
- Solución:
    
    - Confirmalos con `git commit`, o
        
    - Guardalos con `git stash`, o
        
    - Revertilos con `git restore`.
        

---

## En resumen

|Acción|Comando|
|---|---|
|Descargar y fusionar cambios|`git pull origin main`|
|Descargar y reordenar (sin merge)|`git pull --rebase origin main`|
|Solo descargar (sin aplicar)|`git fetch origin main`|
|Resolver conflictos tras pull|editar → `git add` → `git commit`|
