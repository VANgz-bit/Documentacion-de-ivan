
##  ¿Qué es `git push`?

El comando `git push` **envía** los cambios confirmados (commits) desde tu repositorio local hacia el repositorio remoto (como GitHub).

> En otras palabras: actualiza el repositorio remoto con lo que hiciste localmente.

---

##  ¿Qué necesitas antes de hacer `git push`?

Antes de usar `git push`, se deben cumplir estas condiciones:

1. Ya tenés un **repositorio local** inicializado (`git init`).
    
2. Ya hiciste uno o varios **commits** (`git commit`).
    
3. El proyecto está vinculado a un **repositorio remoto** con `git remote add origin URL`.
    
4. Estás ubicado en una **rama válida** (`main`, `dev`, etc.).
    
5. Tenés acceso al repositorio remoto (tu usuario, clave o token, según cómo lo configures).
    

---

#  SINTAXIS GENERAL DE `git push`

```bash
git push <remoto> <rama>
```

### Ejemplo común:

```bash
git push origin main
```

- **`origin`** → Es el nombre del remoto (por defecto).
    
- **`main`** → Es la rama que estás subiendo.
    

---

#  FLUJO TÍPICO DE USO

1. 🔧 Hacés cambios en tus archivos.
    
2. ➕ Usás `git add .` para agregarlos al staging.
    
3. 📝 Hacés `git commit -m "mensaje"` para confirmar.
    
4. ☁️ Finalmente hacés `git push origin main` para subir a GitHub.
    

---

##  Variantes útiles y detalladas

###  1. `git push -u origin main`

```bash
git push -u origin main
```

- `-u` (o `--set-upstream`) le **indica a Git que vincule tu rama local con la del remoto**.
    
- Esto permite que en futuros pushes solo necesites escribir:
    
    ```bash
    git push
    ```
    
- Solo se necesita usar una vez por rama.
    

---

###  2. `git push origin nombre_de_rama`

- Si estás trabajando en otra rama (por ejemplo `dev` o `feature-x`), podés subirla así:
    
    ```bash
    git push origin dev
    ```
    

---

###  3. Subir todas las ramas

```bash
git push --all origin
```

- Sube **todas las ramas locales** al remoto.
    

---

###  4. Subir con tags (versiones)

```bash
git push --tags
```

- Sube los **tags** que hayas creado (por ejemplo, versiones como `v1.0.0`).
    

---

###  5. Forzar un push

```bash
git push --force
```

- **Sobrescribe** lo que hay en el remoto con lo que tenés local.
    
- ⚠️ **Muy peligroso**: borra cambios remotos si no están en tu historial.
    
- Solo se recomienda si sabés lo que estás haciendo (por ejemplo, después de reescribir el historial con `git rebase`).
    

---

##  ¿Qué es `origin`?

Cuando conectás tu repositorio local a GitHub, le ponés un nombre al remoto con este comando:

```bash
git remote add origin https://github.com/usuario/repositorio.git
```

- `origin` es **el alias del repositorio remoto**.
    
- Podrías usar otro nombre (como `github`, `upstream`, etc.), pero por convención se usa `origin`.
    

---

##  ¿Qué es `main`?

- `main` es el **nombre de la rama principal** de tu proyecto.
    
- Antes se llamaba `master`, pero se cambió por cuestiones inclusivas.
    
- Podés tener otras ramas como `dev`, `feature-login`, `hotfix`, etc.
    

---

##  ¿Qué pasa internamente cuando haces `git push`?

Cuando ejecutás `git push`, Git realiza estos pasos:

1. Compara tu rama local con la remota.
    
2. Si tu rama local tiene commits que la remota no tiene, los sube.
    
3. Si hay conflictos (por ejemplo, cambios en la nube que vos no tenés), Git bloquea el push hasta que hagas un `pull` para sincronizar.
    

---

##  ¿Qué errores comunes pueden aparecer?

###  “rejected: failed to push some refs”

Esto ocurre cuando:

- Hay cambios en GitHub que vos no tenés localmente.
    
- Solución:
    
    ```bash
    git pull origin main --rebase
    git push origin main
    ```
    

###  “no upstream branch”

- Ocurre cuando no hiciste el push con `-u`.
    
- Solución:
    
    ```bash
    git push -u origin main
    ```
    

###  “remote: Permission denied”

- Ocurre cuando GitHub no reconoce tus credenciales.
    
- Solución: asegurate de estar autenticado con token o SSH.

|Subir etiquetas|`git push --tags`|
